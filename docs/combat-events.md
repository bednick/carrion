# Событийная модель боя

Бой строится как **типизированный пайплайн-трансформер событий** («стек эффектов»). Это фундамент для системы предметов
и синергий — см. [`content.items.md`](content.items.md) (§4.B теги, §4.F схрон).

> **Статус: реализовано.** Модель живёт в коде: `src/combat/events.ts` (реестр событий), `src/combat/dispatcher.ts`
> (проход/каскад), `src/combat/CombatEngine.ts` (часы стамины + применение терминалов), `src/items/behavior.ts` +
> `src/items/factories.ts` (хуки предметов, `standardWeapon`/`standardArmor`/`standardShield`). Этот документ —
> источник правды и **заменяет** черновой «Пайплайн атаки/защиты» из [`mechanics.md`](mechanics.md).

---

## 1. Принципы

1. **Событие — объект** с `type` и типизированной нагрузкой. `type` определяет, в какие обработчики событие пойдёт;
   нагрузка используется внутри обработчиков.
2. **Обработчик — чистый трансформер.** Принимает одно событие, возвращает результат (см. §4). Состояние **не меняет**,
   UI не трогает.
3. **Состояние меняет только движок.** Он применяет «терминальные» (дошедшие до конца) события к HP и т.п. и реагирует
   на них отрисовкой/звуком. Хуки не знают про рендер.
4. **Персонаж и предметы — одинаковые обработчики.** Персонаж — это просто обработчик с наивысшим приоритетом в общем
   порядке. Никакой отдельной «системы хуков персонажа» нет.
5. **Порядок обработчиков фиксирован:** `character → hand_right → hand_left → head → body → legs → ring → amulet`.
   Обработчик, не подписанный на тип события, пропускается.
6. **Подписка по типу.** Обработчик объявляет методы только на нужные ему типы; отсутствует метод — событие проходит
   без изменений. (В коде — необязательные методы: есть → вызываем.)

## 2. Реестр типов событий

Закрытый перечень, общий для движка/персонажа/предметов. В TS — discriminated union по `type`. Имена — `snake_case`.

```ts
type Side =
    | { side: 'hero'; slot?: SlotType }   // боевой источник: герой (+ каким слотом бьёт)
    | { side: 'enemy'; id: string };

// технический источник: ЧТО породило событие в пайплайне (атрибуция, защита от петель, отладка)
type Origin =
    | { from: 'engine' }                  // системное событие (взмах по таймеру, lifecycle)
    | { from: 'character' }               // породил обработчик персонажа
    | { from: 'item'; slot: SlotType }    // породил предмет в этом слоте
    | { from: 'enemy'; id: string };      // породила защита моба (броня/шипы/уклонение)

interface EventMeta {
    origin: Origin;       // кто эмитнул событие в пайплайн
    cause?: GameEvent;    // событие-причина (родитель в каскаде) — для цепочки/отладки, опционально
}

type GameEvent = EventMeta & (
    // --- жизненный цикл боя ---
    | { type: 'fight_start'; enemies: string[] }
    | { type: 'fight_end'; outcome: 'win' }

    // --- боевые действия ---
    | { type: 'attack_ready'; source: Side; target: Side }                 // часы: поток стамины заполнен, слот готов
    | { type: 'attack'; source: Side; target: Side; amount: number; armorPierce?: number; splash?: boolean }  // взмах, авторённый предметом (урон уже посчитан)
    | { type: 'damage'; source: Side; target: Side; amount: number; armorPierce?: number; splash?: boolean }  // экземпляр урона «в полёте»
    | { type: 'block'; source: Side; target: Side; prevented: number }    // урон полностью отклонён (щит героя / флэт-броня врага в ноль)
    | { type: 'dodge'; source: Side; target: Side }                       // враг уклонился — входящий урон погашен
    | { type: 'counter'; source: Side; target: Side }                     // чисто презентационное: «это был контрудар», HP не трогает
    | { type: 'heal'; source: Side; target: Side; amount: number }       // лечение
    | { type: 'kill'; source: Side; target: Side }                       // цель доведена до 0 HP
    | { type: 'summon'; source: Side; spec: EnemySpec }                  // призыв врага в свободную ячейку доски
    );
```

У всех боевых событий есть `source` и `target` — обязательны, потому что бой бывает с несколькими врагами.
У `summon` `target` нет: цель появления — свободная ячейка доски, её выбирает движок (см. §8). `spec` — уже
**разрешённое** описание противника (`mob_id` → статы, `count` развёрнут в отдельные `summon`), движок не лезет
в конфиги мобов.

**`source` vs `origin`.** `source` — *боевой* источник (кто атакует/лечит, для расчёта). `origin` — *технический*:
какой движок/обработчик породил событие. Их стампит **диспетчер автоматически**: системные события получают
`{ from: 'engine' }`, а `spawn` от обработчика — слот этого обработчика. `cause` (опционально) хранит событие-родителя.
Что это даёт:

- **атрибуция в UI** — «амулет вылечил», «шипы пояса ужалили» (берём слот из `origin`);
- **защита от самопроакта** — предмет может игнорировать события со своим `origin` (не реагировать на собственные
  спавны), поверх общего предохранителя `MAX_CASCADE`;
- **отладка каскада** — по цепочке `cause` восстанавливается, что из чего родилось.

Перечень **расширяемый**: новые типы (`bleed`, …) добавляются сюда же. Любой новый тег-«событие» из §4.B
обязан иметь запись в этом реестре.

### Терминальные события (их применяет движок)

| Тип                       | Что делает движок при применении                                               |
|---------------------------|--------------------------------------------------------------------------------|
| `attack_ready`            | будит автора атаки (хук предмета / дефолт-автор движка); сам HP не трогает     |
| `attack`                  | выводит `damage` из `amount`/`target` (урон уже посчитан автором)              |
| `damage`                  | отнимает `amount` у HP цели; если HP цели → 0, ставит в очередь `kill`         |
| `block`                   | играет анимацию блока, HP не трогает                                           |
| `dodge`                   | играет анимацию уклонения врага, HP не трогает                                 |
| `counter`                 | играет анимацию контрудара (`onCounterAttack`), HP не трогает — спавнится **рядом** с настоящим `attack`, не вместо него (см. `buckler`, `content.items.hand_left.md`) |
| `heal`                    | добавляет `amount` к HP цели (капается `maxHp`)                                |
| `kill`                    | удаляет врага / запускает фазу; если все мертвы — ставит в очередь `fight_end` |
| `summon`                  | строит врага из `spec`, ставит в ближайшую свободную ячейку доски; нет ячеек — гасится |
| `fight_start`/`fight_end` | оркестрация боя в сцене                                                        |

## 3. Результат обработчика: `replace` vs `spawn`

Ключевое различие — два разных намерения, у них разные правила прохода:

```ts
type EventResult = {
    replace?: GameEvent[];  // чем стало ТЕКУЩЕЕ событие; продолжает путь со СЛЕДУЮЩЕГО обработчика
    spawn?: GameEvent[];  // НОВЫЕ события; уходят на СВЕЖИЙ полный проход с начала
};
// отсутствие метода ≡ { replace: [event] } (пропустить без изменений)
```

- **`replace`** — модификация того же события для **остатка** текущей цепочки. Тип обычно сохраняется
  (пример: броня снижает `amount`). Пустой `replace: []` — событие **заглушено**.
- **`spawn`** — порождение нового события, которое должно увидеть **вся доска заново**
  (пример: контрудар, лечение за блок).

**Правило выбора:**
> Меняешь *число/детали* того же события → `replace` (продолжит вперёд, без перезапуска — иначе петля).
> *Превращаешь* в событие другого типа, на которое должны среагировать остальные → `replace: []` + `spawn: [новое]`.

Почему это важно (дыра порядка): щит в `hand_left` отклоняет урон **раньше**, чем «лечение за блок» на `amulet`
успевает его увидеть в том же проходе. Если бы блок был `replace`, `amulet` до него бы просто не дошёл (проход
уже закончился). Поэтому щит делает `replace: []` + `spawn: [block]`, и `block` идёт свежим полным проходом —
`amulet` ловит его и спавнит `heal`.

## 4. Диспетчер: точный флоу

```
process(initialEvent):
  queue = [ stampOrigin(initialEvent, {from:'engine'}) ]
  iterations = 0
  while queue not empty:
    if ++iterations > MAX_CASCADE: warn("event cascade overflow"); break   # предохранитель
    event = queue.shift()
    terminals = runPass(event, queue)     # один полный проход; spawn'ы падают в queue
    for t in terminals:
      engine.apply(t)                     # единственное место смены состояния + реакция UI
                                          # apply может поставить в queue follow-up (attack→damage, damage→kill)

runPass(event, queue):
  handlers = ORDER.filter(h => subscribes(h, *по ходу*))   # character, hand_right, ...
  current = [event]                        # события, ещё «в полёте» в ЭТОМ проходе
  for h in handlers:
    next = []
    for e in current:
      if h.handles(e.type):
        { replace=[e], spawn=[] } = h.run(e, ctx)
        next.push(...replace)              # продолжает к следующему обработчику
        for s in spawn:                    # диспетчер сам проставляет origin = слот обработчика
          queue.push(stampOrigin(s, originOf(h), cause=e))   # свежий проход позже
      else:
        next.push(e)                       # без изменений (тип мог измениться у соседа)
    current = next
  return current                           # терминальные события на применение
```

`stampOrigin`/`cause` проставляет **диспетчер**, а не код предмета: обработчик в `spawn` отдаёт «голое» событие, а
провенанс (слот-источник и событие-причина) навешивается автоматически. Так `origin` нельзя подделать или забыть.

- `replace` продолжает движение **вперёд** по оставшимся обработчикам и **никогда** не перезапускает проход — это
  исключает бесконечную петлю (напр. брони, снижающей урон).
- `spawn` уходит в `queue` и проходит обработчики **с начала** — это и закрывает дыру порядка.
- `MAX_CASCADE` (напр. 64 на одно исходное событие) — страховка от патологических комбинаций. Превышение логируем.
- Внутри прохода тип события может «разойтись» (часть стала `block`) — маршрутизация идёт по **текущему** типу
  каждого события, поэтому подписчик на `block` дальше по порядку его поймает.

### Контекст обработчика (`ctx`)

Только для чтения: текущие HP героя/врагов, **полная экипировка + её теги** (для условных синергий §4.B), `rarity`
обрабатывающего предмета, источник RNG. Сеттеров нет — состояние меняет движок.

## 5. Атаки: стамина → автор → урон

Атака проходит цепочку, где у каждого типа события — **единственное место рождения**:

```
attack_ready  ← движок (часы): поток стамины слота заполнился
   ↓  на него подписан АВТОР
attack        ← автор: у героя — хук предмета слота; у врага — дефолт-автор движка
   ↓
damage        ← движок выводит из терминального attack
   ↓
kill          ← движок выводит из терминального damage при HP→0
```

### Потоки стамины (независимые атаки)

«Сколько независимых атак» — это состояние планировщика движка, а **не** события. Движок держит список потоков:

```ts
type AttackStream = { source: SlotType; interval: number; elapsed: number };
```

- Список строится при старте боя и на смене экипировки; **длина = число независимых атак** (можно показать в UI как N
  полосок стамины).
- Интервал движок берёт у предмета через `attackInterval(rarity)` (см. ниже), а не из сырого конфига.
- Поток заполнился (`elapsed ≥ interval`) → движок эмитит `attack_ready{ source:{hero, slot} }` и сбрасывает `elapsed`.
  Урон движок при этом не считает.

### Автор атаки (статы — в скрипте)

Combat-статы и их скейл по редкости живут в `behavior.ts` — это единый источник правды. Предмет-оружие выставляет
интерфейс:

```ts
interface WeaponBehavior {
    attackInterval(rarity: Rarity): number;   // ЧАСЫ: движок зовёт при сборке потока; light здесь делит на scale
    onAttackReady(e, ctx): EventResult;        // АВТОР: считает урон и спавнит attack(и)
}
```

- На `attack_ready` срабатывает хук предмета своего слота и **спавнит** `attack{ amount }` с уже посчитанным (
  отскейленным) уроном. Это единственное место, где рождается `attack` героя.
- **Мультихит** — автор на один `attack_ready` спавнит **два** `attack` (без спецполя `hits_per_fill`). Два честных
  `attack` → два `damage` → лайфстил прокает дважды.
- **Враг** предметов не имеет: его `attack` авторит дефолт-автор движка по `attacks[]` моба. Конвейер один, различается
  только автор.

### Всё в behavior.ts

Отдельного `config.json` нет: `behavior.ts` — единственный файл предмета (кроме `icon.svg`) и единственный источник
правды и по идентичности/экономике (`name`, `slots`, `type`, `baseValue`, `tags` — обязательные поля `ItemBehavior`),
и по боевой математике (база статов, скейл по редкости, авторство атаки, защитные хуки).

Чтобы простые предметы не разрастались — **фабрики**: число и кривая скейла идут аргументами, а идентичность
дописывается поверх спредом.

```ts
const behavior: ItemBehavior = {
  name: 'Тяжёлое оружие', slots: ['hand_right'], type: 'weapon', baseValue: 10, tags: ['weapon', 'heavy'],
  // тяжёлое: урон растёт с редкостью, интервал фиксирован
  ...standardWeapon({damage: 5, interval: 2.0, scale: {damage: 1.6, interval: 1.0}}),
};
```

`standardWeapon` возвращает только боевую часть (`ItemCombatBehavior` — `ItemBehavior` без identity-полей); особые
оружия (мультихит, cleave, условный урон) пишутся руками поверх. Аналогично `standardArmor`/`standardShield` для
`damage_reduction`/`block_chance`, которые в событийной модели тоже становятся хуками (§6).

Следствия: **один источник правды** (тултип/скупщик зовут те же `attackInterval`/расчёт урона — числа не разъезжаются с
боем); движок больше **не интроспектит** сырые `attack_damage`/`type==='weapon'`; атаковать может не только рука — любой
предмет, выставивший поток.

## 6. Как выражаются эффекты (включая текущие)

| Эффект                       | Подписка → результат                                                |
|------------------------------|---------------------------------------------------------------------|
| Броня (`damage_reduction`)   | на `damage`(target=hero): `replace:[damage с меньшим amount]`       |
| Щит (`block_chance`)         | на `damage`(target=hero): шанс → `replace:[]` + `spawn:[block]`     |
| Лечение за убийство (амулет) | на `kill`: `spawn:[heal(target=hero)]`                              |
| Лайфстил за удар             | на `damage`(source=hero, target=enemy), кроме `splash: true`: `spawn:[heal(target=hero)]` |
| Контрудар (`buckler`)        | на `damage`(target=hero), независимый ролл: `spawn:[attack(source.slot=hand_right, target=атакующий враг) + counter]` — `attack`, не `damage` напрямую, чтобы поймать крит-хуки вроде `heavy_gloves`; `counter` — чисто презентационный дубль рядом (см. §2, `content.items.hand_left.md`) |
| Шипы (`spiked_shield`)       | на `damage`(target=hero), безусловно: `spawn:[damage(target=атакующий враг)]` (доля дошедшего урона) |
| Броня врага (`armor`)        | на `damage`(target=enemy): `replace:[damage с меньшим amount]`      |
| Уклонение врага              | на `damage`(target=enemy): шанс → `replace:[]` + `spawn:[dodge]`    |
| Шипы врага                   | на `damage`(target=enemy): `spawn:[damage(target=hero)]`           |

### Защита врага (обработчик моба)

Враг — обработчик наравне с предметами (§1.4), но предметов не носит: его защита — **данные из конфига моба**
(`MobConfig.defense: { armor?, dodge?, thorns? }`, см. [`content.mobs.format.md`](content.mobs.format.md)).
Встроенный хук бежит **первым** в проходе на `damage`(target=этот враг) — до слотов героя, чтобы уклонение сняло
удар **до** он-хит эффектов героя (лайфстил не прокает на уклонённый удар), а броня отдала им реально прошедший
урон. Порядок внутри хука: **dodge → armor → thorns** (уклонился — броня/шипы не применяются). Спавны штампуются
`origin: { from: 'enemy', id }`. Ничего отдельно не скейлится — значения берутся как есть.

Броня (`armor`) — процентная и мультипликативная (0..1, с 2026-07, см. `docs/mechanics.md` §«Броня vs щит»,
`src/combat/mitigation.ts`): снижает `amount`, но никогда не обнуляет удар целиком — округление гарантирует
минимум 1, если исходный урон был > 0. `armorPierce` удара (напр. крит `war_pick`, см.
`content.items.hand_right.md`) снижает эффективную броню для конкретного удара. Терминальное `block`(target=enemy)
от брони больше не спавнится (не может сработать при мультипликативной модели, ветка убрана вместе с
FloaterType `absorb`/надписью «Отражено», ныне мёртвым UI) — единственная реакция врага на входящий удар с
собственной надписью теперь `dodge` → UI «мимо».

Эмерджентность бесплатна: «лайфстил за удар» автоматически сильнее с `fast`-оружием — больше срабатываний `damage`,
без спец-кода на пару (§4.B).

`splash` (0/1, аналогично `armorPierce` по механике переноса) — булев флаг на побочных тиках урона у cleave-оружия
(`broadaxe`): основной удар по цели атаки его не несёт, все дополнительные удары по прочим врагам — несут. Движок
переносит поле с `attack` на `damage` так же, как `armorPierce`. Он-хит эффекты героя (`leech_bead`) явно
игнорируют `splash: true`, чтобы один взмах по толпе не лечил героя за каждую задетую цель.

### Барьер и аварийный хил героя (не хуки предметов)

`amulet` может дать герою барьер (временный HP-пул на бой) и аварийный хил (разово при HP ниже порога) — см.
[`content.items.amulet.md`](content.items.amulet.md). В отличие от всего в §6 выше, это **не** `on`-хуки: обоим
нужно мутируемое состояние, которое живёт **весь бой** (тающий пул барьера, флаг «хил уже использован»), а
обычный хук — честная функция `(event, ctx) => result` без памяти между вызовами. Поэтому оба реализованы как
движковые примитивы уровня `hero.hp`, симметрично защите врага выше, только для героя и в терминальной точке, а
не в `runPass`:

- **Грант** читается один раз из `ItemBehavior.barrierAmount`/`emergencyHeal` при сборке героя
  (`CombatEngine.buildInitialHero`, тем же приёмом, что `attackInterval` → `buildWeaponTimers`).
- **Срабатывание** — не проход по `HANDLER_ORDER`, а прямая логика в `CombatEngine.applyDamage` (ветка
  `target === 'hero'`), **после** того как `runPass` уже прогнал все item-хуки (броня/щит/шипы). Барьер поглощает
  первым остаток урона, дошедший до этой точки; аварийный хил проверяется после того, как остаток вычтен из
  `hp`. Никакого нового `EventType` для этого не заводили.
- Барьер сбрасывается (полный грант заново, без переноса) на каждый **новый бой** зоны, аварийный хил — флаг
  «использован» сбрасывается там же, но не при смене фазы **одного** боя (воскрешение Нежити не даёт вторую
  попытку в той же битве).

### Соответствие текущему коду (миграция)

| Сейчас в `CombatEngine`                                 | Станет                                                     |
|---------------------------------------------------------|------------------------------------------------------------|
| `buildWeaponTimers` (читает `type`/`interval`/`damage`) | поток из `attackInterval()` + хук-автор `onAttackReady`    |
| `attack_damage`/`attack_interval` в `config.json`       | в `behavior.ts` (фабрика `standardWeapon`)                 |
| `ItemBehavior.onKill`                                   | обработчик на `kill`                                       |
| `ItemBehavior.onTakeDamage` (цепочка)                   | обработчик на `damage`(target=hero), `replace`             |
| блок щита (инлайн в `applyEnemyAttack`)                 | `standardShield` → обработчик на `damage`, `spawn:[block]` |
| снижение брони (инлайн)                                 | `standardArmor` → обработчик на `damage`, `replace`        |
| колбэки `onDamageDealt`/`onBlock`/…                     | реакция движка на терминальные события (UI-слой)           |

## 7. Пример сквозного прохода

Враг бьёт героя на 5. Надето: щит (`hand_left`, блок) + гипотетический амулет «лечение за блок» (`amulet`) —
**чисто иллюстративный** пример механики прохода `spawn`/`replace`, не реальный/запланированный предмет:
«лечение за блок» рассмотрено и отклонено для `amulet` (см. [`content.items.amulet.md`](content.items.amulet.md)
§«Отклонённые направления» — дублирует по эффекту броню/митигацию).

```
queue: [ damage{src:enemy#1, tgt:hero, amount:5} ]

проход 1 (damage):
  character           — не подписан → пропуск
  hand_right          — не подписан → пропуск
  hand_left (щит)     — блок выпал → replace:[]  spawn:[ block{src:hero/hand_left, tgt:hero, prevented:5} ]
  → damage заглушён; block добавлен в очередь с origin={item, hand_left}, cause=damage
terminals: []   (HP не изменилось)

queue: [ block{...} ]

проход 2 (block):
  character           — пропуск
  hand_right          — пропуск
  hand_left           — пропуск
  head                — пропуск
  body                — пропуск
  legs                — пропуск
  ring                — пропуск
  amulet (амулет)     — spawn:[ heal{tgt:hero, amount:2} ];  replace:[block] (block летит дальше для отрисовки)
                        → heal в очередь с origin={item, amulet}, cause=block
terminals: [ block ]      → движок играет анимацию блока (атрибуция: блок от hand_left)

queue: [ heal{...} ]

проход 3 (heal):
  никто не подписан → terminals: [ heal ]   → движок лечит героя на 2 (капается maxHp)

queue пуст → стоп
```

## 8. Заметки

- **Чувствительность к порядку.** Для `replace`-трансформаций результат зависит от порядка слотов (A влияет на B,
  только если A раньше). Для `spawn` это не так (свежий проход видят все). Дизайня синергию, помни про §1.5.
- **RNG.** Блок/крит внутри хуков используют RNG — для воспроизводимости в тестах источник случайности должен быть
  внедряемым (часть `ctx`), а не глобальный `Math.random`.
- **Несколько врагов и доска.** Бой идёт на доске из **4 фиксированных ячеек** в один ряд; позиция врага задаётся
  ячейкой (`slot`), а не индексом в массиве. При генерации `attack_ready` движок целит героя во врага в **крайней
  левой занятой ячейке** (ближайшей к герою) и прокидывает цель автору. `summon` сажает нового врага в ближайшую
  свободную ячейку от призывателя; нет свободных — призыв гасится (это и есть лимит доски). См.
  [`content.mobs.format.md`](content.mobs.format.md) — раздел «Призыв».
