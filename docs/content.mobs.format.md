# Формат мобов

Все поля в json — snake_case (например `mob_id`, а не `mobID`; `item_id`, а не `itemId`).

Файл `src/mobs/{mob_id}/config.json`. Через этот формат задаётся **любой** противник — и рядовой моб, и босс.
Боссовость — это не отдельный тип, а лишь финальный бой зоны (см. [формат зон](content.zones.format.md)).

```json
{
  "id": "...",
  "name": "...",
  "health": 0,
  "attacks": [
    {
      "damage": 1,
      "interval": 1
    },
    {
      "damage": 1,
      "interval": 2
    }
  ]
}
```

- `attacks` — массив независимых атак. Каждая со своим уроном и интервалом (в секундах), свой таймер.
  Несколько элементов = мультиатака (несколько независимых полосок атаки). Обязателен хотя бы один.
- Необязательное поле `scale: { min, max }`.
- Необязательное поле `ui` — визуальные настройки отображения, не влияют на боевую логику.
  - `ui.alpha` (0..1, по умолчанию `1`) — прозрачность спрайта моба. Например `"ui": { "alpha": 0.5 }` для призрачных мобов.
  - `ui.scale` (по умолчанию `1`) — множитель к итоговому размеру спрайта. Спрайт моба сперва вписывается
    без искажений в стандартный бокс (150×130 для рядовых, 210×165 для боссов), а `ui.scale` домножает
    результат этого вписывания — например, `0.6` для мобов, которые должны казаться заметно мельче соседей
    (мышь, крыса), `1.3` — заметно крупнее. Якорь остаётся на линии тени, так что моб растёт/уменьшается
    вверх от земли, не смещая ноги.
- `position` (по умолчанию `0`) — ячейка доски (0..3), в которую встаёт этот моб как **корневой** в бою.
  `0` — крайняя левая (ближайшая к герою). У призванных мобов это поле не действует — их ячейку определяет призыватель.
- `summon_placement` (по умолчанию `nearest`) — стратегия расстановки его призывов, см. ниже.
- **Лут не задаётся в мобе.** Добыча привязана к зоне: общий лут рядовых мобов — `mob_loot`, награда за
  босса — `boss.loot` (см. [формат зон](content.zones.format.md)). Это позволяет переиспользовать одного моба
  в разных зонах с разной добычей.

## Защита — `defense` (необязательно)

Хук на входящий урон по этому мобу. Все поля опциональны, применяются в порядке **dodge → armor → thorns**.
Ничего отдельно не скейлится — значения берутся как есть.

```jsonc
"defense": {
  "armor": 0.3,    // доля 0..1: мультипликативно снижает каждый входящий удар, никогда не обнуляя целиком
  "dodge": 0.3,    // шанс 0..1 полностью погасить входящий удар
  "thorns": 2      // отразить фикс. урон обратно в героя за каждый полученный удар
}
```

- `armor` — **не блок**: сильнее давит мелкий чип, большому одиночному пробою почти не мешает (относительное
  снижение % в обоих случаях одинаковое, но в абсолютных числах слабый удар просаживается заметнее). Награждает
  тяжёлый одноцелевой билд, наказывает быстрый мультихит (каждый удар мультиатаки снижается отдельно). Никогда
  не сводит удар в ноль — округление гарантирует минимум 1 урон, если исходный удар был > 0 (см.
  `src/combat/mitigation.ts`).
- `dodge` — при удаче удар гасится целиком (событие `dodge` для UI); броня/шипы в этот удар не применяются.
- `thorns` — отражает урон в героя за сам факт удара (даже после снижения бронёй); на уклонённый — нет.

Дизайн-роли фракций — см. [`factions.md`](factions.md). Формат события — [`combat-events.md`](combat-events.md) §6.

## Призыв — `summons` (необязательно)

Моб может призывать дополнительных противников — в начале боя, периодически, по порогу HP или **при смерти**. Призыв
ссылается на другого моба по `mob_id` и берёт его статы; поля статов можно переопределить (`name`, `health`, `attacks`).
У каждой записи **обязателен** `trigger` — он задаёт, *когда* срабатывает призыв.

```jsonc
"summons": [
  { "mob_id": "henchman", "count": 1, "trigger": { "type": "start" } },
  { "mob_id": "henchman", "trigger": { "type": "interval", "every": 6.0 } },
  { "mob_id": "rat", "count": 2, "trigger": { "type": "hp", "at": 0.5 } },
  { "mob_id": "bone_stub", "count": 2, "trigger": { "type": "death" } }
]
```

### `trigger.type`

| `type`     | Когда призывает                                    | Доп. поля        |
|------------|----------------------------------------------------|------------------|
| `start`    | один раз в начале боя                              | —                |
| `interval` | каждые `every` секунд, пока призыватель жив        | `every` (сек)    |
| `hp`       | один раз, когда HP призывателя падает ≤ `at`·maxHp | `at` (доля 0..1) |
| `death`    | один раз, когда призыватель окончательно гибнет    | —                |

`count` (по умолчанию `1`) — сколько противников появляется за одно срабатывание.

**`death` vs `hp: { at: 0 }`.** Для «моб рассыпается на N врагов при смерти» бери `death`. Триггер `hp` при
`at: 0` **не сработает**: движок не опрашивает планы призыва у моба с `hp ≤ 0` (мёртвых пропускает тик), а сама
смерть идёт через отдельный путь `kill`. `death` срабатывает именно в этом пути — **после** проверки фаз (если у
моба есть `phases` и ролл в форму прошёл, `death`-призыв не сработает: моб не «умер», а сменил форму). Призванные
садятся от ячейки павшего — она к этому моменту уже свободна, поэтому первый занимает её место, остальные —
ближайшие свободные.

### Куда встаёт призыв — `position` и `summon_placement`

Ячейка для призванного моба выбирается так:

1. Если у записи призыва задан **`position`** (0..3) — это приоритетная целевая ячейка. Занята? Берётся ближайшая
   свободная к ней.
2. Иначе — по стратегии **`summon_placement`** призывателя (поле корневого моба):

| `summon_placement` | Опорная ячейка призыва                                       |
|--------------------|--------------------------------------------------------------|
| `nearest`          | ближайшая свободная **от самого призывателя** (по умолчанию) |
| `front`            | ближайшая свободная **к герою** (крайняя левая, слот 0)      |
| `back`             | дальняя от героя (крайняя правая, слот 3)                    |

```jsonc
"summon_placement": "back",
"summons": [
  { "mob_id": "henchman", "trigger": { "type": "start" } },            // по стратегии моба → в тыл
  { "mob_id": "shield", "position": 0, "trigger": { "type": "start" } } // явно в слот 0 (щит перед боссом)
]
```

### Доска и лимит

Бой идёт на **доске из 4 фиксированных ячеек** (слотов) в один ряд. Корневой моб встаёт в свою `position` (по
умолчанию — крайняя левая, слот 0); призванные занимают ячейку по правилам выше. Смерть противника освобождает
ячейку — её может занять следующий призыв.

**Лимит — это сами слоты:** если свободных ячеек нет, призыв молча пропускается. Отдельного `max_active` или кулдауна
нет — потолок доски (4) и так ограничивает число противников.

Герой всегда атакует противника в **крайней левой занятой ячейке** (ближайшего к себе), поэтому фланговые призванные
прикрывают босса — их приходится прорубать первыми.

## Фазы — `phases` (необязательно)

Моб может при «смерти» перейти в следующую форму. `phases` — список форм после базовой, по порядку.
Каждая форма ссылается на моба по `id` и может частично переопределить его описание.

```jsonc
"phases": [
  { "id": "headless_knight" }
]
```

`chance` (0..1, по умолчанию `1`) — шанс входа в форму. Фазы проверяются **по порядку при смерти текущей формы**:
если ролл прошёл — моб переходит в форму и продолжает бой; если **не** прошёл — цепочка обрывается и моб умирает
окончательно (последующие фазы не проверяются).

```jsonc
// три формы по 50%: P(дойти до 2-й) = 0.5, до 3-й = 0.25
"phases": [
  { "id": "form_a", "chance": 0.5 },
  { "id": "form_b", "chance": 0.5 },
  { "id": "form_c", "chance": 0.5 }
]
```

Разворачивается **один уровень**: если моб-форма сам объявляет `phases`, они игнорируются (рекурсии нет).
