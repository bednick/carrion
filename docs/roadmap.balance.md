# Баланс — план продолжения (после итерации с предметами)

Снимок состояния на момент паузы: сделан первый проход баланса **трёх стартовых зон** (★☆☆). Дальше работа
переключается на **создание предметов** для средних/финальной зон — этот файл фиксирует, что нужно знать и
сделать, когда вернёмся к балансу.

> **Как пользоваться.** Не дублирует `content.md` §«Баланс — здоровье и опорный УВС» (там канон правила и текущие
> цифры трёх зон) — здесь только то, что нужно **сделать дальше** и контекст, который иначе придётся
> реконструировать заново.

## Что уже сделано (не трогать без причины)

- **Правило зафиксировано** в `docs/content.md` §«Баланс — здоровье и опорный УВС»: HP-тиры (вводный 25 / обычный
  50 / сильный 75 / босс 100 за фазу), число боёв по группам зон, формула УВС, целевой остаток HP.
- **Число боёв закреплено во всех 9 реальных зонах** (`fights.min/max` в `src/zones/*/config.json`):
  старт 5–7, люди/звери 9–11, нежить 13–15 (`battlefield` без `mob_pool` — не считово).
- **Три стартовые зоны (`dead-fields`, `trampled-meadows`, `armor-dump`) полностью пересчитаны и провалидированы**
  Monte-Carlo симулятором (`npm run balance`) — финальные `health`/`attacks[].damage` в их `src/mobs/*/config.json`
  и синхронной прозе `docs/zones/{зона}.md`.
- **Калибровочный якорь для стартовых зон — безоружный игрок**, не формульные 50%/75%. Билд
  `tools/builds/fresh-start.json` (`battle_staff` `common`, без брони/щита — это реальная стартовая экипировка,
  `src/core/MetaStore.ts` → `defaultStands()`) — основной ориентир. Результат: winrate 88–100%, HP перед боссом
  60–72%, после победы 11–35% («на волоске», по задаче).
- **Побочный эффект, сознательно принятый:** билд `tools/builds/basic-uncommon.json` в этих трёх зонах теперь
  почти без риска (броня гасит урон подчистую). Если это будущая проблема — см. открытый вопрос ниже.

## Что НЕ сделано и должно быть следующей итерацией

### 1. Предметы для 6 средних зон + финала — предпосылка баланса

Сейчас реальные предметы существуют **только** для трёх стартовых зон (12 предметов, потолок редкости
`uncommon`, слоты `hand_right`/`hand_left`/`body`/частично `legs`, см. `docs/content.items.md` §«Охват
текущей проработки» — `head`, `ring`, `amulet` домены ещё не спроектированы). Без предметов средних зон баланс
там нельзя ни просчитать (нет реального `hero_DPS_ref`), ни провалидировать (`npm run balance` строит билд из
`ItemInstance[]`, взятых из реальных `src/items/*`).

Порядок действий, когда вернёмся:
1. Спроектировать и завести предметы средних зон (`legs`, `hand_right`, `hand_left`, `body`; редкость до
   `epic` — потолок для не-стартовых зон, см. `content.md` §«Редкость предметов»). Использовать скилл `content`
   для мобов/зон уже не нужен — тут скорее ручная работа по `docs/content.items*.md` + `src/items/factories.ts`.
2. Собрать 2–3 репрезентативных билда (`tools/builds/*.json`) под новый ассортимент — по аналогии с
   `basic-balanced.json`/`basic-uncommon.json`, но для тира средних зон.
3. Только после этого пересчитывать `health`/`attacks[].damage` мобов средних зон.

### 2. Присвоить HP-тир каждому мобу 6 средних зон

Зоны: `abandoned-camp`, `marauder-lair`, `beast-lair`, `crypt`, `mage-ruins`, `predator-pasture`. Способ — по
весу в `mob_pool` (топ вес → тир `обычный`, низкий вес → тир `сильный`; тира `вводный` в средних зонах не бывает,
см. правило). Ниже — флейвор-теги, собранные разведкой по `docs/zones/*.md` в этой же сессии (нельзя терять —
дорого реконструировать заново). Модификаторы см. в `content.md` — они описывают ОТКЛОНЕНИЕ HP от базового тира
роли, не абсолютные числа.

**`abandoned-camp`** (боевые мобы-люди, вес 20→8): `veteran`(20, база, «бьёт размеренно и сильно, держит

### 3. Открытый вопрос: якорь калибровки для средних зон

Для старта якорем стал «гарантированный минимум» (`battle_staff` без брони). Для средних зон такого
единственного гарантированного билда нет — игрок приходит с чем-то, найденным/скрафченным в стартовых зонах,
диапазон широкий. Варианты на выбор при возврате к задаче:
- Взять формульные 50%/75% как есть (изначальный план) — самый простой путь, уже описан в `content.md`.
- Определить «минимальный правдоподобный билд при входе в среднюю зону» (например: один `common`-предмет
  каждого найденного стартового слота) и добавить его как второй якорь, как это сделали для старта.

Не решено — уточнить с пользователем перед пересчётом чисел средних зон, не предполагать самостоятельно
(см. `[[feedback_balance_placeholders]]` в памяти — здесь та же ловушка: не полагаться на цифры, которые выглядят
осмысленными, если они не провалидированы диалогом/симулятором).

### 4. `battlefield.md` — отдельный контентный гэп, не баланс

Финальный босс «Дитя битвы» и рядовые «Осколок битвы»/«Сшитый зверь» полностью спроектированы в
`docs/zones/battlefield.md` (числа, фазы, призывы), но **мобов нет в `src/mobs/`**, а `src/zones/battlefield/
config.json` ссылается на `skeleton` как боссовую заглушку. Это создание нового контента (см. скилл `content`,
формат — `docs/content.mobs.format.md`/`docs/content.zones.format.md`), не пересчёт чисел — делать отдельным
заходом, вне этого баланс-роадмапа.

### 5. Инструменты, которые уже есть — не изобретать заново

- `npm run balance -- <zoneId> --trials=5000 --build=<path>` — Monte-Carlo по всей экспедиции зоны
  (`src/balance/simulate.ts`, общий код с `balance.html`).
- `tools/builds/fresh-start.json` — билд нового игрока (только `battle_staff` `common`).
- `tools/builds/basic-uncommon.json`, `basic-balanced.json`, `weapon-armor.json`, `weapon-only.json` — прочие
  тестовые сборки, все на предметах стартовых зон; под средние зоны нужны новые файлы после появления предметов.

### 6. Эмпирические уроки этой итерации — не наступать на те же грабли

- **Плоская броня режет мелкий чип в ноль** (`content.mobs.format.md` §«Защита»): формула УВС без поправки на
  типичную броню игрока систематически даёт заниженный урон мобов. Всегда проверять сырые числа против брони
  тира зоны, не только против голой формулы.
- **Своры/несколько тел резко повышают дисперсию** относительно одиночного моба той же средней DPS — при равном
  среднем остатке HP смертность может быть на порядок выше. Для мобов с `summons: [{ trigger: start }]`
  закладывать это заранее, не только после первого прогона симулятора.
- **Числа в прозе `docs/zones/*.md` до сих пор не гарантированно синхронны с `src/mobs/*/config.json`** для
  6 нетронутых зон — они там плейсхолдер, как и было изначально (`10/5/2.0`), не путать с уже пересчитанными
  тремя стартовыми.
