# Планируемый функционал (не реализовано)

Единый список механик, **задуманных в контенте/дизайне, но ещё не поддержанных движком**. Сюда попадает то, что
описано в `docs/` или заведено в конфигах «на вырост», но пока не работает в бою — чтобы не терялось и чтобы при
реализации был один источник правды.

> **Как пользоваться.** Прежде чем «просто добавить моба/предмет с новой механикой» — проверь, есть ли она здесь.
> Если механики нет в движке, контент заводится с плейсхолдер-поведением (обычно — простая атака по дефолтам), а сюда
> добавляется запись. Реализовал — **удали запись** отсюда и обнови профильный док (`combat-events.md` и т.д.).

> **Рефактор петли (2026-07) выполнен.** 3 стойки-пресета, крафт=улучшение/разбор, один герой (Силач),
> центр boss-only с гейтом на 9 зон, гарант-драфт после босса, авто-лут в сундук (нет рюкзака/схрона/побега).
> Формат меты несовместим — при первом запуске **сбросить прогресс** (кнопка «Сброс данных» в лагере или
> очистка `localStorage` `carrion.meta.v1`).

## Формат записи

- **Механика** — короткое имя.
- **Откуда пришло** — контент/док, который её требует.
- **Что нужно в движке** — на каком уровне встраивается (событие/статус/хук).
- **Плейсхолдер сейчас** — как соответствующий контент ведёт себя до реализации.

---

> **Защита врага реализована (2026-07).** Флэт-броня, уклонение (шансовое) и шипы врага живут в движке через
> `MobConfig.defense: { armor?, dodge?, thorns? }` — встроенный хук на `damage`(target=enemy), порядок
> dodge → armor → thorns (см. [`combat-events.md`](combat-events.md) §6, [`content.mobs.format.md`](content.mobs.format.md)).
> Демо-мобы: `buried_shade` (dodge), `armored_veteran` (armor), `spiked_veteran` (thorns). Остальным
> бронированным/шипастым/уклоняющимся мобам зон нужно лишь **проставить числа** `defense` по зональным докам —
> это контент, не движок. Оконный/фазовый вариант уклонения (бестелесная фаза во времени) не делали: он ждёт
> «Статусы с длительностью» ниже; шансового варианта хватает.

## Доска боя 2×2 (позиционирование и формы сплеша)

- **Откуда пришло:** семейство сплеш-оружия `hand_right` — см. [`content.items.hand_right.md`](content.items.hand_right.md).
  Разные формы сплеша (удар по линии, прошив по колонне, удар по соседним) обретают смысл только на 2D-доске;
  на линейном ряду они сливаются.
- **Что нужно в движке:** передел доски из линейного ряда (4 ячейки) в **сетку 2×2**:
  - позиция врага = (ряд × колонна), а не индекс в ряду;
  - **правило таргета героя** заново (фронт раньше тыла? какая колонна?) вместо «крайняя левая занятая»;
  - размещение `summon` в сетке;
  - поведение **босса** (одна ячейка / целый ряд / весь грид — открыто);
  - **multi-target attack**: один `attack_ready` → `attack` по нескольким ячейкам (cleave/прошив);
  - рендер и [`combat-events.md`](combat-events.md) §8 («4 ячейки в один ряд») переписать.
- **Плейсхолдер сейчас:** доска линейна (1 ряд × 4). Оружие проектируется **под линейный ряд**; сплеш-архетипы
  (cleave/прошив/рикошет) помечены в `content.items.hand_right.md` как ждущие этой доски и не реализованы.

## Система стаков (заряды / накопление)

- **Откуда пришло:** оружейные архетипы `hand_right` — **раскачка урона** (ramp: стаки на цели) и **стан-шкала**
  (накопление → порог → стан); потенциально комбо-счётчики и заряды. См.
  [`content.items.hand_right.md`](content.items.hand_right.md).
- **Что нужно:** **отдельная проработка системы целиком.** При старте — заново собрать требования и описать
  архитектуру отдельным доком. Примитив: счётчик стаков с источником (герой / предмет / **цель**), порогом
  срабатывания, распадом во времени и событиями накопления/сброса поверх событийной модели. Пересекается со
  «Статусами с длительностью» (ниже) — стан и ramp живут на враге во времени.
- **Плейсхолдер сейчас:** стак-оружие (ramp / стан) не реализовано, помечено roadmap; стаков в бою нет.
- **Цеп (`flail`, ramp) — отложен (2026-07).** Вынесен из активной проработки `content.items.hand_right.md`
  (там прорабатывались только предметы с готовым или дешёвым движком) до системы стаков. Задумка на будущее:
  урон растёт за серию ударов по **одной** цели (`ramp_step` за удар, кап `ramp_cap`), **сброс при смене
  цели** — наказывает вязкого одиночку, против своры не работает (цель меняется), само-балансится под
  Мародёров. Плейсхолдер-числа из черновика: урон 2, инт. 1.6, +1/удар, cap +6, скейл стака ×1.4; tags
  `weapon,balanced,ramp`.

## Статус-проклятие (debuff на героя)

- **Откуда пришло:** моб **Плакальщик** (`wailer`) и босс **Бальзамировщик** (`embalmer`), зона Склеп —
  см. [`docs/zones/crypt.md`](zones/crypt.md).
- **Что нужно в движке:** система **статусов с длительностью** на герое + событие наложения/снятия. Проклятие
  ослабляет героя на время: снижает исходящий урон **или** глушит лечение. В событийной модели это обработчик,
  который висит, пока активен статус, и модифицирует `damage`(source=hero) / `heal`(target=hero). Требует:
  (1) тип-событие `curse`/`apply_status` в реестре [`combat-events.md`](combat-events.md) §2;
  (2) хранилище активных статусов и их тиков во времени в `CombatEngine`.
- **Плейсхолдер сейчас:** `wailer` бьёт одной слабой обычной атакой; `embalmer` — двумя обычными полосками
  (мультиатака) + фаза, без проклятия.

## Статусы с длительностью (общий фундамент)

- **Откуда пришло:** предпосылка для «Статус-проклятия» выше и **оконного** уклонения врага (бестелесная фаза во
  времени — шансовое уклонение уже сделано без неё); также нужна предметам Склепа
  с эффектом во времени — **Саван-оберег** (реген HP посекундно), см. [`docs/zones/crypt.md`](zones/crypt.md) §Предметы.
- **Что нужно в движке:** общий механизм временных эффектов (баффы/дебаффы/периодические тики) поверх событийной
  модели — часы статусов в `CombatEngine`, события наложения/истечения, применение per-tick (`heal`/`damage`/
  модификаторы). Сейчас модель обрабатывает только мгновенные каскады событий; понятия «эффект, живущий N секунд» нет.
- **Плейсхолдер сейчас:** таких предметов/эффектов в игре ещё нет (предметы Склепа не заведены).
