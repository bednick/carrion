# Контент: восстановление (слот `amulet`)

Источник правды по слоту `amulet`: домен, ось, характеристики предмета, список предметов и (для барьера/
аварийного хила) движковый механизм, на котором они держатся. Общие правила системы предметов (R1–R9) и
функциональность слотов — [`content.items.md`](content.items.md); боевая идентичность фракций —
[`factions.md`](factions.md); событийная модель — [`combat-events.md`](combat-events.md).

## Домен слота

**Восстановление HP-ресурса героя.** Раньше домен планировался на `body` — решение пересмотрено: весь хил переехал
в `amulet` целиком, `body` отвечает только за снижение получаемого урона
([`content.items.body.md`](content.items.body.md)). Граница между слотами держится жёстко: `amulet` не снижает
входящий урон численно (это `body`/`hand_left`) — он либо возвращает HP, либо даёт герою отдельный ресурс поверх
HP (барьер).

## Ось слота и решения этой правки

Два легаси-предмета уже в коде — `leech_bead` (лечение % от нанесённого урона, on_hit) и `vulture_amulet`
(лечение за килл) — оба на **наступательном** триггере (реагируют на урон, который наносит герой). Эта правка
закрывает наступательный триггер для **нового** контента: новых предметов, влияющих на исходящий урон, слот
больше не получает. `leech_bead`/`vulture_amulet` остаются как есть и подлежат обычному балансу чисел наравне с
остальными — не удаляются и не переосмысливаются.

Новые направления — только **защитные**, и только **косвенно**: реагируют на то, что происходит с героем, но не
через добавление стата снижения урона (это была бы прямая защита, домен `body`), а через HP-ресурс:

- **Аварийный хил** — реактивный, по состоянию героя (HP ниже порога), не более раза за бой.
- **Барьер** — проактивный, по времени (открытие боя), отдельный пул поверх HP.

Единой continuum-оси между этими двумя и легаси-парой в этой правке не вводится: три разных механических формы
(offensive-trigger лечение, threshold-triggered лечение, fight-start ресурс) описаны как открытый список, не как
жёсткий полюс-полюс (так же решили для `body`).

| Предмет                                      | Триггер                 | Профильный стат       | Природа                                    |
|----------------------------------------------|-------------------------|-----------------------|--------------------------------------------|
| Пиявочная бусина (`leech_bead`, легаси)      | урон, нанесённый героем | % лечения от урона    | наступательный, закрыт для нового контента |
| Амулет грифа (`vulture_amulet`, легаси) | килл                    | % макс. HP убитого врага | наступательный, закрыт для нового контента |
| Оберег на волоске (`thread_charm`, новый)    | HP героя < порога       | величина хила         | защитный, реактивный, 1 раз за бой         |
| Амулет барьера (`barrier_amulet`, новый)     | начало боя              | размер барьера        | защитный, проактивный, весь бой            |

## Отклонённые направления

- **Лечение за блок** — рассмотрено (был готовый пример в `combat-events.md` §7), отклонено: не берём.
- **Лечение за полученный урон** (безусловное) — отклонено: по эффекту неотличимо от брони (снижает эффективный
  урон на ту же величину через другой канал), дублирует домен `body`, а не дополняет его.
- **Пассивный реген по времени** (тик, не привязан ни к одному событию) — не берём в этой итерации: в движке нет
  тикающего вне таймера оружия триггера, событийная модель полностью event-driven. Кандидат на будущее.
- **Наступательный триггер для новых предметов** — закрыт этой правкой (см. выше). Существующие
  `leech_bead`/`vulture_amulet` — легаси-исключение, не прецедент для новых предметов.

## Барьер — механика (проработана как основание для будущих предметов)

Не item-хук, а **движковый примитив уровня HP**: барьер — новое поле состояния героя
(`HeroState.barrier`/`barrierMax`), а не что-то, что предмет считает сам на каждое событие. Причина: барьеру
нужно **мутируемое состояние, которое живёт весь бой** (тает по мере поглощения ударов) — у обычных item-хуков
такого нет, они честные `(event, ctx) => result` без памяти между вызовами. Тот же принцип уже применялся к
`hero.hp` — барьер устроен симметрично.

- **Грант.** `ItemBehavior.barrierAmount?: (rarity) => number` — декларативное значение, читается один раз при
  сборке героя (`CombatEngine.buildInitialHero`, новая функция `buildHeroResources`, тот же приём, что и у
  `attackInterval`/`buildWeaponTimers`). Суммируется по всей надетой экипировке (на практике — один амулет), не
  жёстко привязано к слоту `amulet`.
- **Выдача.** Полный `barrierMax` — на **каждый** бой (`beginFight()` в `ExpeditionScene` строит нового героя
  через `buildInitialHero` при входе в следующий бой зоны). В отличие от HP (который явно переносится между
  боями забега — «накопительный сток», см. `roadmap.balance.md`), барьер **не переносится**: каждый новый бой
  начинается с полного пула, независимо от того, сколько осталось от предыдущего.
- **Поглощение.** Место — `CombatEngine.applyDamage`, ветка `target === 'hero'`, **до** вычитания из `hero.hp`:
  `absorbed = min(hero.barrier, amount); hero.barrier -= absorbed; amount -= absorbed`. Остаток (если есть) идёт
  в HP как обычно. Порядок в пайплайне: барьер стоит **последним** — `applyDamage` вызывается уже после полного
  прохода `runPass` (щит/броня/шипы уже отработали как item-хуки), поэтому барьер поглощает **финальный** урон
  после всей обычной митигации, а не сырой удар. Это соответствует жанровому паттерну «щит после брони»
  (Diablo/Destiny/Overwatch), не наоборот.
- **UI.** Новый колбэк `CombatCallbacks.onBarrierAbsorb?(amount)`, презентационный (не трогает `damage`-путь) —
  всплывающий текст «−N барьер» (новый `FloaterType: 'barrier'`) + тонкая полоска над HP-баром
  (`ExpeditionScene.heroBarrierFill`), видна только пока `barrier > 0`.
- **Расширяемость («мы будем добавлять новые механики поверх него»).** Барьер — состояние движка, не спрятанное
  в замыкании одного предмета, поэтому будущие эффекты могут читать текущий уровень барьера
  (`CombatView.heroBarrier`/`heroBarrierMax` — стоит завести по аналогии с `heroHp`/`heroMaxHp`, если/когда
  появится первый потребитель) или гранить барьер из другого источника (килл, оверхил-конвертация и т.п.) —
  сам пул уже общий ресурс, не собственность одного предмета.

### Открытые вопросы барьера

1. **Хот-свап экипировки посреди боя** (`ExpeditionScene.applyEquipToHero`) пересобирает героя целиком
   (`buildInitialHero`) и явно переносит `hp`, но **не** переносит `barrier` — значит, смена амулета посреди боя
   полностью обнуляет/переполняет барьер до нового максимума. Для HP это осознанно иначе (перенос), для барьера
   пока не решали отдельно — вероятно, ту же логику переноса стоит применить (`fresh.barrier = min(engine.state.
   hero.barrier, fresh.barrierMax)`), но это отдельное микрорешение, не блокирующее первую итерацию.
2. **`CombatView` пока не содержит `heroBarrier`** — добавить, когда появится первый предмет/эффект, которому
   нужно читать текущий барьер (YAGNI: не заводить прежде первого потребителя).

## Аварийный хил — механика

Тоже движковый примитив, не item-хук — по той же причине: нужен флаг «уже сработал в этом бою»
(`HeroState.emergencyHealUsed`), мутируемый между несколькими проходами `damage`.

- **Конфиг.** `ItemBehavior.emergencyHeal?: (rarity) => { thresholdRatio, healPercent }` — декларативно, читается
  один раз в `buildInitialHero` (та же `buildHeroResources`). `thresholdRatio` **не скейлится редкостью**
  (фиксирован на 30%, R5 — один профильный стат на архетип, им выступает `healPercent`). `healPercent` — доля
  от `heroMaxHp` (0..1), не флэт-число: сам хил считается от `hero.maxHp` **в момент срабатывания**
  (`applyDamage`), а не один раз при сборке героя — если maxHp когда-нибудь станет зависеть от экипировки/статов,
  хил останется корректным без правки этой механики.
- **Срабатывание.** `CombatEngine.applyDamage`, после того как урон (за вычетом барьера) вычтен из `hero.hp`:
  если герой жив, эффект ещё не использован в этом бою и `hp / maxHp < thresholdRatio` — лечит на
  `round(maxHp * healPercent)` (капается по `maxHp`, как обычный `heal`) и ставит `emergencyHealUsed = true`.
  Флаг сбрасывается каждый новый бой (`buildInitialHero`), не при смене фазы одного боя (воскрешение Нежити не
  даёт второй заряд в той же битве).
- **Почему «раз за бой», а не пороговый без лимита.** Без ограничения эффект стабильно доливал бы HP на
  **каждом** ударе, пока герой ниже порога — фактически становясь ещё одной формой мягкой брони, только выраженной
  через хил, и переставая быть «аварийным». Лимит держит его редким и заметным, как и задумано.
- **UI.** Переиспользует существующий `onHeal`-канал (тот же флоатер/обновление HP-бара, что у `leech_bead`/
  `vulture_amulet`) — отдельного нового колбэка не потребовалось.

## Предметы

| Предмет           | id               | Профильный стат                | tags                          | База (common) | По редкости (common/uncommon/rare/epic) | Статус              |
|-------------------|------------------|--------------------------------|-------------------------------|---------------|-----------------------------------------|---------------------|
| Пиявочная бусина  | `leech_bead`     | шанс лечения за удар (флэт HP)  | `accessory,lifesteal,on_hit`  | 20% шанс, +1 HP | 20% / 30% / 40% / 50%                 | легаси, ребаланс (вероятностный прок) |
| Амулет грифа | `vulture_amulet` | % макс. HP убитого врага        | `accessory,on_kill,lifesteal` | хил 20% HP    | 20% / 30% / 40% / 50%                   | легаси, ребаланс    |
| Амулет барьера    | `barrier_amulet` | `barrierAmount`                 | `accessory,barrier`           | барьер 4      | 4 / 6 / 9 / 14                          | новый, в коде       |
| Оберег на волоске | `thread_charm`   | `healPercent` (порог фикс. 30%) | `accessory,last_stand`        | хил 25% HP    | 25% / 40% / 55% / 70%                   | новый, в коде       |

`barrier_amulet` — по аналогии со `scaleByRarity(base, rarity, 1.5)`, как у большинства предметов `body`/
`hand_left`. `thread_charm`/`vulture_amulet`/`leech_bead` — линейная прогрессия (+10/+15 п.п. за тир), не
экспоненциальная: подобрана вручную под процентную семантику (экспонента дала бы слишком резкий разброс между
тирами). `vulture_amulet` перешёл с фикс. числа на % от `maxHp` убитого врага (был плоским — не масштабировался под
жирных мобов/боссов, теперь пропорционален их HP); `leech_bead` перешёл с капирования лечения по величине урона
(`scaleByRarity(1, rarity, 1.5)`) на вероятностный прок фиксированной величины (+1 HP), тот же паттерн `ctx.rng()
>= CHANCE[ctx.rarity]`, что у `heavy_gloves`/`buckler`/`heavy_shield`; `legendary` у всех трёх не крафтится —
повторяет значение `epic` (`HEAL_PERCENT`/`PROC_CHANCE` в соответствующих `behavior.ts`), как
`desperate_plate.REDUCTION`. Все наборы чисел **не проверены Monte-Carlo** — прогнать `npm run balance`, когда
появятся репрезентативные билды.

## Реализовано в коде (2026-07)

- Новый тег `barrier` заведён в реестр тегов `content.items.md`; `last_stand` переиспользован из `body`
  (`desperate_plate`) — тот же смысл «эффект при HP ниже порога», просто на другом слоте и с другим эффектом
  (хил вместо брони).
- `ItemBehavior.barrierAmount`/`emergencyHeal` (`src/items/behavior.ts`) — декларативные поля, читаются
  `buildHeroResources` в `CombatEngine.ts`.
- `HeroState.barrier`/`barrierMax`/`emergencyHeal`/`emergencyHealUsed` (`src/combat/types.ts`), заполняются в
  `CombatEngine.buildInitialHero`.
- Поглощение барьера и аварийный хил — в `CombatEngine.applyDamage` (ветка `target === 'hero'`), без изменений в
  `dispatcher.ts`/`runPass`/новых `EventType` — оба живут ПОСЛЕ полного прохода item-хуков, на уровне терминальной
  аппликации урона, там же, где и `hero.hp -= amount`.
- `CombatCallbacks.onBarrierAbsorb` + `FloaterType: 'barrier'` (`src/core/EventBus.ts`, `src/ui/Floater.ts`) +
  тонкая полоска барьера над HP-баром (`ExpeditionScene.heroBarrierFill`/`updateHeroHpBar`).
- Предметы `barrier_amulet`, `thread_charm` — `src/items/*`, зарегистрированы (`registry.ts`, `icons.ts`),
  добавлены в лут-таблицы зон рядом с `leech_bead`/`vulture_amulet` (конкретная раскладка — только
  `src/zones/*/config.json`, см. `content.items.md` §R11).
- Сборка (`npm run build`, `tsc`) чистая.

## Открытые вопросы

1. **Числа не Monte-Carlo-проверены** (см. «Предметы») — прогнать `npm run balance`, когда появится
   репрезентативный билд с амулетом.
2. **Хот-свап барьера посреди боя** — см. «Открытые вопросы барьера» №1.
3. ~~Дом предметов `amulet` в лут-таблицах~~ — **снято (2026-07, R11).** Конкретная зона-источник для каждого
   предмета этим доком больше не отслеживается, см. `src/zones/*/config.json`. Слот `amulet` формально
   внефракционен (не участвует в R10 как «чемпион одной фракции»).
4. **Судьба легаси-пары.** `leech_bead`/`vulture_amulet` остаются наступательными по триггеру. `leech_bead`
   переведён (2026-07) с гарантированного лечения (капированного величиной удара) на вероятностный прок фикс.
   величины — снимает жалобу на «слишком сильно на common», где эффект срабатывал на каждом ударе. `vulture_amulet`
   пока не тронут — числа в коде (`HEAL_PERCENT: 10/15/20/25/25%`) расходятся с таблицей выше
   (`20/30/40/50%`), актуализировать отдельно.
